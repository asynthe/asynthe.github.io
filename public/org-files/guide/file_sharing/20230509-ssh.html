what is SSH

What is SSH?: It means Secure SHell, and it's used to connect to remote Linux or [BROKEN LINK: 1c0db150-cde1-453f-8912-cc641fc11a51] servers, although it can also work with Windows servers.

It is intended to send and receive secure and encrypted information between two machines: the local machine and a remote machine.

There are two programs that are needed for this to work

  • The SSH Server (a daemon in the remote machine, always running and listening through an TCP/IP port for a connection)
  • The SSH Client (local machine, if it has the correct credentials, it can connect to the remote machine)

You can use SSH KEYS to authenticate your session, making it more secure than using passwords everytime.

flags

flags for ssh:

-o -> IdentitiesOnly=yes/no -> -i ->

example: using -o "IdentitiesOnly" and -i will tell SSH to only use specified key. $ ssh -i <private-key-filename> -> Use the default key (id_ed25519.pub, id_rsa.pub) and take that private as an extra. $ ssh -o "IdentitiesOnly=yes" -i <private-key-filename> -> Use that private key as the only key.

flags for ssh-keygen:

-t -> specify encryption -C "email@domain.com" ->

flags for ssh-copy-id:

-i / identity_file -> if you're specifying another key from the default (~/.ssh/id_ed25519.pub, ~/.ssh/id_rsa.pub) -f -> force keys, don't check if they are already installed -p -> port -n -> dry run, no keys are copied -s -> use sftp, uses sftp instead of remote-commands.

note: Adding a passphrase makes it more secure, without it + disabling password authentication, you can log in to a remote machine without any trouble. The trouble comes when someone realizes you have that kind of setup.

Create a key on your local machine. $ ssh-keygen -t ed25519 add a secure passphrase too

Send the password through SSH to the remote machine $ ssh-copy-id <remote-user>@<remote-ip>

Log into the machine typing your passphrase $ ssh <remote-user>@<remote-ip>

DISABLING PASSWORD AUTHENTICATION

In the remote machine, edit /etc/ssh/sshd_config, and uncomment the next line: 

# To disable tunneled clear text passwords, change to no here!
#PasswordAuthentication yes

to 

# To disable tunneled clear text passwords, change to no here!
PasswordAuthentication no

Reboot the SSH daemon in the remote machine # systemctl reload sshd

PASSWORDLESS AUTHENTICATION

How to Setup Passwordless SSH Login - Linuxize How to force ssh to use a specific private key? - StackExchange

  1. Make a ssh key without a passphrase.

    $ ssh-keygen -t ed25519

    If a custom file is specified example: /home/user/debiankey In this directory (/home/user), two files will be created

    • A private key, debiankey
    • A public key, debiankey.pub

  2. Import this passphrase to the server.

    $ ssh-copy-id <server-user>@<server-ip> or, with a custom key (make sure it's the public, stupid) $ ssh-copy-id -i ~~/temp/ssh/debiankey.pub <server-user>@<server-ip>

    In case the ssh-copy-id is not available, you can use the following command. $ cat ~~/.ssh/id_ed25519 | ssh <server-user>@<server-ip> "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"

  3. Logging into the server with the SSH keys

    Now, you log in with SSH, it shouldn't ask for a password now. $ ssh <server-user>@<server-ip>

    note: This uses your default keys found in ~/.ssh/id_ed25519.pub or ~/.ssh/id_rsa.pub. To specify a key in another directory or file, use: $ ssh -o "IdentitiesOnly=yes" -i ~~/temp/ssh/debiankey <server-user>@<server-ip> remember, it's the private key in this case.

  4. OPTIONAL: Disable SSH Password Authentication on the Server

    CAUTION: Make sure all the steps above work for you, and that you user has sudo privileges.

    This can add a layer of security to your server, as passwords will be disabled and the only way to connect will be through the aforementioned SSH keys.

    1. Log into the remote server with SSH (root / user with sudo privileges)

    2. On the server, edit /etc/ssh/sshd_config as it follows: 

      PasswordAuthentication no
ChallengeResponseAuthentication no
UsePAM no

      Once done, restart the service # systemctl restart ssh CentOS or Fedora: # systemctl restart sshd

    3. Done, try loggin in with password and with keys.
